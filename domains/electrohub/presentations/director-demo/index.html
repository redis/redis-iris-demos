Context Surfaces Model

ElectroHub Ships A Real Context Surfaces Data Model

from context_surfaces.context_model import (
    ContextField,
    ContextModel,
)


class Product(ContextModel):
    __redis_key_template__ = "electrohub_product:{product_id}"

    product_id: str = ContextField(
        description="Unique product identifier",
        is_key_component=True,
    )
    sku: str = ContextField(
        description="Retail SKU used in checkout and store systems",
        index="tag",
        no_stem=True,
    )
    name: str = ContextField(
        description="Customer-facing product name",
        index="text",
        weight=2.2,
    )
    category: str = ContextField(
        description="Top-level category such as Computers or Audio",
        index="tag",
    )
    form_factor: str = ContextField(
        description="Device shape such as laptop, desktop, or mini_pc",
        index="tag",
    )
    price: float = ContextField(
        description="Current selling price",
        index="numeric",
        sortable=True,
    )
    specs_summary: str = ContextField(
        description="Readable hardware summary for the product",
        index="text",
    )
    search_text: str = ContextField(
        description="Expanded searchable text for product discovery",
        index="text",
        weight=1.8,
    )

Why this matters

  • This is the Context Surfaces model the ElectroHub app actually registers.
  • Products, orders, shipments, support cases, and inventory each have explicit indexed fields.
  • Context Surfaces can turn that model into practical tools like search_product_by_text and filter_storeinventory_by_product_id.

Tool Discovery

The App Discovers And Calls Context Surfaces Tools At Runtime

async def list_tools(self) -> list[dict[str, Any]]:
    async with UnifiedClient() as client:
        tools = await client.list_tools(self.settings.mcp_agent_key)
    return [t if isinstance(t, dict) else t.model_dump() for t in tools]


async def call_tool(self, tool_name: str, arguments: dict[str, Any]) -> dict[str, Any]:
    async with UnifiedClient() as client:
        result = await client.query_tool(
            agent_key=self.settings.mcp_agent_key,
            tool_name=tool_name,
            arguments=arguments,
        )

ElectroHub examples

  • search_product_by_text for product discovery
  • filter_storeinventory_by_product_id for pickup truth
  • filter_order_by_customer_id for order history
  • filter_shipment_by_order_id and shipment events for late-delivery answers
Agent Execution

LangGraph Uses Those Tools To Answer Multi-Step Commerce Questions

async def create_agent(settings, internal_tools, cs_service, checkpointer=None):
    model = ChatOpenAI(...)

    tools = _make_internal_tools(internal_tools)
    mcp_defs = await cs_service.list_tools()
    tools.extend(_make_mcp_tool(t, cs_service) for t in mcp_defs)

    return create_react_agent(
        model,
        tools,
        prompt=prompt,
        checkpointer=checkpointer,
    )

Why this works for ElectroHub

  • The model can plan, call product and inventory tools, inspect the results, and keep going.
  • That fits retail questions because one answer often spans multiple entities and data sources.
  • Examples: product + inventory, or order + shipment + shipment events.
Context Surfaces

Where Redis Context Surfaces Changes The Game

Without it

  • Handwritten adapters for every entity.
  • Prompt drift around tool names and query semantics.
  • More surface area to maintain for every new demo.

With it

  • Structured Redis models become MCP tools.
  • The LLM sees filter_* and search_* tools automatically.
  • Same operational data plane supports agentic reasoning and retrieval.

What improves

  • Less custom glue.
  • Clearer operational answers.
  • Better separation between retrieval and live state.

In ElectroHub, the same agent can reason over products, store inventory, shipments, and support cases because Context Surfaces exposes all of those entities consistently as MCP tools.

One Platform, Multiple Agent Needs

Vector retrieval Guides and policies for the Simple RAG path
Checkpoint memory Conversation state stored in Redis for the LangGraph agent
Operational truth Redis JSON / indexed entities surfaced as MCP tools
Knowledge retrieval Vector search over policy and guide content
Agent memory Redis-backed LangGraph checkpointing across turns
Conversation State

ElectroHub Uses langgraph-checkpoint-redis

from langgraph.checkpoint.redis.aio import AsyncRedisSaver


checkpointer = AsyncRedisSaver(
    redis_url=redis_url,
    checkpoint_prefix=domain.manifest.namespace.checkpoint_prefix,
    checkpoint_write_prefix=domain.manifest.namespace.checkpoint_write_prefix,
)
await checkpointer.asetup()

agent = create_react_agent(
    model,
    tools,
    prompt=prompt,
    checkpointer=checkpointer,
)

Why it helps

  • AsyncRedisSaver persists LangGraph thread state in Redis.
  • Conversation state lives on the backend, not only in the browser.
  • Follow-up questions can reuse prior context and tool history by thread.
  • It makes demos easier to debug because state is durable and inspectable.
